# Skill: Weekly Consolidation — Sessions -> Knowledge Topics

**HARD GATE — read first:**

> When this skill triggers because a bucket is due (session number % 5 == 0),
> NO other work is allowed until (a) consolidation is done and
> (b) `scripts/build_knowledge_index.py` runs with no `broken_refs`/`duplicates`/`missing-frontmatter`.
> No feature coding, no debugging, no answering unrelated questions before that.

**When to trigger:** after writing `sessions/session_NNN.md` when `NNN % 5 == 0`
(i.e. after session 5, 10, 15, 20, ...). The SessionStart hook reminds you.

**Goal:** distill raw sessions into structured knowledge topics under `knowledge/`.
Raw sessions stay untouched.

---

## Target layout
```
knowledge/
├── INDEX.json                <- generated by scripts/build_knowledge_index.py
├── architecture/             <- ARCH-NNN_*.md
├── tools/                    <- TOOL-NNN_*.md
├── projects/<project>/       <- PROJ-NNN_*.md + progress.md per project
├── processes/                <- PROC-NNN_*.md
├── learnings/                <- L-NNN_*.md   (error fixes)
├── decisions/                <- D-NNN_*.md   (ADR style)
└── rules/                    <- RULE-NNN_*.md
```

## Frontmatter schema (required for every .md)
```yaml
---
id: L-014
title: "Node fetch r.body is not a Web ReadableStream"
category: learnings
status: active        # active | deprecated | superseded
related: [TOOL-003, L-007]
sessions: ["028", "029"]
tags: [node, fetch, stream]
last_verified: 2026-06-03
---
```

The `id` prefix MUST match the directory: `ARCH`->architecture, `TOOL`->tools, `PROJ`->projects,
`PROC`->processes, `L`->learnings, `D`->decisions, `RULE`->rules. The index validates this.

---

## Step by step

### 1. Read sessions N-4 .. N
```bash
N=30   # current session number (adjust)
for i in $(seq $((N-4)) $N); do
  printf "%03d ---\n" "$i"
  cat "sessions/session_$(printf %03d $i).md" 2>/dev/null || echo "(missing)"
done
```
Note what is new, what changes existing topics, what are fixes/ADRs.

### 2. Find existing topics + free IDs
```bash
for p in ARCH TOOL PROJ PROC L D RULE; do
  echo -n "$p next -> "
  ls knowledge/*/ 2>/dev/null | grep -oE "${p}-[0-9]+" | sort -V | tail -1 \
    | awk -F- -v p=$p '{printf "%s-%03d\n", p, $2+1}'
done
```
If none exist -> start at `<PREFIX>-001`.

### 3. Category mapping
| Sessions contain ... | -> write to / update |
|---|---|
| New architecture design (components/data flow) | `knowledge/architecture/ARCH-NNN_<slug>.md` |
| New/changed tool insight (CLI, API wrapper, script) | `knowledge/tools/TOOL-NNN_<slug>.md` |
| Project progress | `knowledge/projects/<project>/progress.md` (append) + maybe `PROJ-NNN_<feature>.md` |
| Repeatable workflow / process | `knowledge/processes/PROC-NNN_<slug>.md` |
| **Error + fix + lesson** (even small ones!) | `knowledge/learnings/L-NNN_<slug>.md` |
| Architecture decision with rationale | `knowledge/decisions/D-NNN_<slug>.md` (ADR style) |
| Universal rule (holds across sessions) | `knowledge/rules/RULE-NNN_<slug>.md` |

### 4. Required body sections
**learnings/L-NNN_*.md:** `## Error` · `## Root Cause` · `## Fix` · `## Lesson` (one generalized line) · `## Fallout` (set `related:`)
**decisions/D-NNN_*.md (ADR):** `## Context` · `## Options` · `## Decision` · `## Consequences` · `## Status`

### 5. Extend existing topic files (do NOT overwrite)
1. Update frontmatter: bump `last_verified:`, extend `sessions:`.
2. Append a new section at the END: `## Update YYYY-MM-DD (Session NNN)`.
3. NEVER delete existing sections — on contradiction, mark the old state ~~struck through~~ and add the new.

### 6. Cross-refs in BOTH directions
If `L-014` relates to `TOOL-003`, put `related: [TOOL-003]` in L-014 AND `related: [..., L-014]` in TOOL-003.
Otherwise the index flags `broken_refs` (or `orphans`) on the next build.

### 7. Build the index + verify
```bash
python3 scripts/build_knowledge_index.py
```
Output must show: `0 broken refs, 0 duplicates, 0 missing-frontmatter`. Orphans are okay (e.g. standalone RULEs).
On errors:
```bash
python3 -c "import json; d=json.load(open('knowledge/INDEX.json')); \
  print('broken:', d['broken_refs']); print('dup:', d['duplicates']); print('missing:', d['missing_frontmatter'])"
```
-> fix, rebuild, until green.

### 8. Update `memory.md`
- "Current state": keywords + pointers into new topic IDs (`see ARCH-007, L-018`).
- Session index table: row added for the last 5 sessions?
- **No duplication** of topic content — only pointer + one line of context.

### 9. Sync auto-memory (`memory/MEMORY.md` + files)
- For new `RULE-*` or high-relevance `L-*` -> one bullet in `memory/MEMORY.md` with a pointer into the topic file.
- Stale auto-memory files: delete or migrate to a topic, clean the index.

### 9b. Curate the agentic-loop list (required — `memory/AGENTIC_LOOP.md`)

This file is injected at EVERY session start (hook). It is the self-improvement layer for
**working method / turn efficiency** — not for project facts. At every consolidation:

1. Scan sessions N-4..N for **recurring friction**: where were turns/tokens lost? Rabbit holes?
   Duplicate work? Wrong assumptions? Knowledge that existed but wasn't recalled?
2. Record every *generalizable* lesson as **"Trigger -> do this" + (evidence/session)**.
   Working method only; project-specific facts go to memories/topics instead.
3. **Trim:** max ~8 entries. Remove internalized/no-longer-relevant points. Review the
   candidates section — promote what got confirmed.
4. **Orphan check** (applies here too): every new/changed `memory/` file needs a `MEMORY.md`
   pointer (command in `end_of_session.md` step 5b).

### 9c. Curate skill proposals (required — `skills/SKILL_PROPOSALS.md`, the 3x rule)

After every consolidation, check the fresh learnings/sessions for **recurring workflows** that
deserve to become a skill (multi-step, cross-project or cross-task, was re-derived ad hoc):
1. New pattern -> add an entry to `skills/SKILL_PROPOSALS.md` (counter = 1, date + evidence sessions).
2. Existing proposal shows up again -> **counter +1** with date + new evidence.
3. **Counter reaches 3 -> the skill gets created** (`skills/<name>.md` or `.claude/skills/<name>/`),
   set the ledger status to `CREATED: <path>`, mention it in the summary to the user.
4. Proposals no longer relevant -> set to `rejected: <reason>`, never delete (history).

### 10. Git commit
```bash
git add knowledge/ memory.md memory/ skills/SKILL_PROPOSALS.md
git commit -m "consolidation: sessions $((N-4))-$N -> X new topics (Y learnings, Z decisions)"
```
Don't switch branches. No `--amend`.

---

## Anti-patterns (do NOT do)
- Overwrite topic files instead of extending them.
- Set `related:` one-directionally.
- Commit with `git add -A` (the PreToolUse hook blocks it anyway) — always stage explicitly.
- Hand-edit INDEX.json — it is ALWAYS regenerated.
- Abort the skill without a green index build.

## Quality check before commit
1. `python3 scripts/build_knowledge_index.py` -> exit 0, no broken_refs/duplicates/missing.
2. Every new topic has ≥1 `related:` OR is legitimately standalone (RULE).
3. `sessions/session_<N-4..N>.md` unchanged (original protection).
4. `memory.md` not bloated, no topic content duplicated.
5. Git status: only `knowledge/`, `memory.md`, `memory/`, `skills/SKILL_PROPOSALS.md` modified.
